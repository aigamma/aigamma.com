# Site audit, 2026-05-13

Source pass run by Claude Code while Eric was traveling. Five parallel
investigation agents ran (frontend perf, netlify functions, bundle and
CSS, dead code and stale references, Supabase data layer), and a sixth
crawl walked every MCP connection (Netlify, Supabase, GitHub, Gmail,
Hugging Face). This doc separates what was already fixed in-session
from what needs Eric to decide before any further change, and closes
with a feature-ideas section for surfaces that would help readers
uncover market inefficiencies.

## Light fixes already applied in this session

All committed and pushed to main on 2026-05-13. Each commit message
carries the full rationale; this list is the cliff notes.

- **CLAUDE.md per-page-prompt list rewrite.** The Source-of-Truth Map
  section listed `smile,parity` in the chat-prompt enumeration, but
  both files were retired weeks ago (parity 2026-05-07, smile
  2026-05-08). The list now reads `main, garch, regime, rough, local,
  jump, risk, discrete, tactical` and matches the on-disk reality the
  page-consistency check already validates.

- **`US_MARKET_HOLIDAYS` centralized to a shared module.** The set of
  29 NYSE holiday dates spanning 2026 through 2028 was duplicated
  verbatim across seven Netlify functions (eod, expiring-gamma,
  ingest, ingest-background, narrate, reconcile, reconcile-
  background). All seven now import from
  `netlify/functions/lib/market-calendar.mjs`, so the 2029 refresh is
  one edit instead of seven. The new module also exports
  `isTradingDay` and `previousTradingDay` helpers that replace inline
  weekday loops in the same functions.

- **EOD per-ticker fan-out parallelized.** `phaseVixFamily` and
  `phaseDailyEod` in `eod-downsample-background.mjs` were running 52
  sequential Massive aggregates calls (17 VIX-family + 35 stocks),
  adding 15 to 25 seconds per EOD cycle for nothing. Both phases now
  route through a `fetchTickerBatch` helper that uses
  `Promise.allSettled` with a concurrency window of 5. Tolerant of
  individual failures (the previous "warn and continue" semantic
  preserved); ceiling is intentionally conservative and can be lowered
  if Massive starts rate-limiting.

- **`narrate-background.mjs` regex compilation and JSON pretty-print.**
  Hoisted the four regexes used by `sanitizeNarratorString` and
  `parseNarratorOutput` to module-level constants (compiled once
  instead of once per narrator output), and removed the `null, 2`
  pretty-print argument from the `JSON.stringify(state)` call that
  feeds the user message into Anthropic. The model sees the same
  structure either way; pretty-printing inflates input-token count
  ~3x for nested objects with no semantic payoff.

- **`rag_documents` table comment in Supabase rewritten to match the
  actual ingest walker.** The previous comment claimed CLAUDE.md and
  docs/ were in the corpus; the walker (`scripts/rag/ingest.mjs`) has
  never traversed either. The new comment names exactly what's
  indexed (per-page prompts + the three global blocks + about
  subdomain) and points at the SOURCES array as the extension point.

## Recommendations needing Eric's call

These need a decision before applying — either a Supabase schema
migration that should not be cowboy-applied to prod, or a structural
choice that affects how the chat or data layers behave.

### Supabase security, MEDIUM priority

1. **Revoke `EXECUTE` on `cleanup_partial_snapshots_v1` and
   `cleanup_success_snapshots_v1` from `anon` and `authenticated`.**
   Both are SECURITY DEFINER and reachable at `/rest/v1/rpc/...`
   without authentication. An attacker who sends
   `?p_age_days=0` would delete every row in `snapshots` (16M
   rows, the entire intraday history). The intended caller is
   `snapshots-retention.mjs` using the service key, which bypasses
   the grant. Single migration:

   ```sql
   REVOKE EXECUTE ON FUNCTION
     public.cleanup_partial_snapshots_v1(integer),
     public.cleanup_success_snapshots_v1(integer)
     FROM anon, authenticated;
   ```

2. **Drop the `anon_insert_*` RLS policies on `computed_levels`,
   `expiration_metrics`, `ingest_runs`, `snapshots`, and the
   `anon_update_svi_fits` policy on `svi_fits`.** Each policy uses
   `WITH CHECK (true)` and effectively grants the anon role write
   access to the intraday persistence layer. The intended writer is
   `ingest-background.mjs` with the service key. Whoever wrote the
   policies originally was likely trying to test inserts from a local
   client and forgot to remove the temporary grant; the result is
   that anyone with the publishable anon key (which ships in every
   chat call to the frontend) could write a forged `ingest_runs` row
   pointing to a forged `snapshots` payload and pollute downstream
   computations. This is the most consequential single finding from
   the whole audit.

3. **Set `search_path` on the four PL/pgSQL functions flagged by the
   linter** (`set_updated_at_timestamp`, `check_rate_limit`,
   `get_system_prompts`, `match_rag_chunks`). Lower urgency because
   no untrusted users have CREATE in `public`, but trivially
   hardened with one migration:

   ```sql
   ALTER FUNCTION public.set_updated_at_timestamp() SET search_path = public, pg_temp;
   ALTER FUNCTION public.check_rate_limit(text, text, integer) SET search_path = public, pg_temp;
   ALTER FUNCTION public.get_system_prompts(text) SET search_path = public, pg_temp;
   ALTER FUNCTION public.match_rag_chunks(vector, text, integer) SET search_path = public, pg_temp;
   ```

### Supabase data hygiene, LOW priority

4. **Prune the 38 orphaned `/smile/` rows from `page_narratives`.**
   `/smile/` was retired 2026-05-08 with a 301 to `/jump/`, but the
   narrator continued writing rows for it until then; 38 rows remain
   with `last_seen = 2026-05-08 21:57:34`. No reader queries those
   rows because `pages.js` has no `/smile/` entry, so they're inert,
   but they're dead weight in a table that the narrator is filling
   at ~1400 rows/day. One-line cleanup:

   ```sql
   DELETE FROM page_narratives WHERE page = '/smile/';
   ```

5. **Pick a retention policy for `page_narratives`.** The narrator is
   writing roughly 1400 rows per day at steady state; at that rate
   the table hits ~500K rows in a year. No retention RPC exists. A
   30-day window is probably the right default (the table is a
   diagnostic history, not an authoritative record), but anywhere
   from 14 to 90 days is defensible. A `cleanup_page_narratives_v1`
   function modeled on the existing snapshots-retention RPC plus a
   Sunday-morning cron is the implementation.

6. **Investigate the 67% partial-rate on `ingest_runs` from the last
   7 days.** 281 partial / 133 success / 11 error across the recent
   window, with reconciliation `partial_rate` and `run_count` failing
   every trading day. The 15000ms FETCH_TIMEOUT_MS in
   `ingest-background.mjs` is being hit on Massive's options chain
   pagination; the error message "no contracts or no spot price"
   masks what's actually a connection-or-rate-limit failure. The
   weekly retention RPC is supposed to clean up partial-run snapshots
   but rows from 2026-04-14 are still present. Worth: (a)
   instrumenting the error path in `ingest-background.mjs` to
   distinguish timeout from empty response and log the underlying
   Massive HTTP code, and (b) verifying the
   `snapshots-retention.mjs` cron is actually firing on Sundays.

### RAG corpus, LOW priority

7. **Extend `scripts/rag/ingest.mjs` to walk `docs/` and `CLAUDE.md`,
   or leave the corpus as-is.** The table comment in Supabase has
   been corrected to match reality (only the 13 source paths are
   actually ingested today). The question is whether the chatbot
   would benefit from being able to retrieve from
   `docs/rag-architecture.md`, `docs/options-volume-roster.md`, and
   `docs/earnings-data-roadmap.md`. Those documents do contain
   reader-relevant material (RAG topology, options-volume threshold
   logic, earnings data roadmap) that the chatbot currently can't
   reach. Cost: a moderate walker extension plus one ingest pass.
   Benefit: chatbot can answer "how does the RAG layer work" or
   "what's the planned market-cap data source" without falling
   back to similarity hits on tangentially-related prompt chunks.

### Frontend, LOW priority

8. **Wrap Chat.jsx token-stream `setMessages` calls in
   `startTransition`.** During an active chat the SSE deltas trigger
   one `setMessages` call per token. React 18 batching softens this
   but the message-list re-render still fires dozens of times per
   second. `startTransition(() => setMessages(...))` deprioritizes
   the token arrivals and lets React batch / interrupt them under
   load. The chat already works smoothly so this is a
   diminishing-returns optimization, but it's the most-mentioned
   item the frontend audit kept surfacing.

9. **Add `useCallback` to the three pointer handlers in
   `RangeBrush.jsx` and memoize the five style objects.** Drag
   interactions fire at 60 Hz pointer events; each currently
   recomputes the handler functions and the style objects from
   scratch. Smooths the brush feel on slower devices. The component
   is mounted on multiple charts (the main dashboard brush, the
   SpxVolFlip brush, the DealerGammaRegime brush), so the fix
   propagates.

### Bundle / CSS, LOW priority

10. **Move repeating inline `style={{...}}` patterns in `src/App.jsx`
    into CSS classes.** Roughly 95 inline-style instances on the
    homepage that fall into a handful of repeating shapes (`color:
    var(--text-primary)` 35x, `color: var(--accent-coral)` 17x,
    `margin: '0 0 0.7rem'` 7x). Extracting to CSS classes shaves a
    few kilobytes off the main bundle and lets the browser cache
    them across visits. Pure refactor with no visual change.

11. **Split `src/styles/page.css` into per-feature CSS files.** The
    file is 93KB unminified, shipped to every non-homepage entry,
    and contains large feature-specific blocks (econ-events ~1000
    LOC for `/beta/` only, earnings ~320 LOC, seasonality ~400 LOC,
    heatmap ~110 LOC, scan ~100 LOC). Extracting to per-page
    imports would save 15-20KB per page entry for pages that don't
    use those features. Ambitious refactor: touches 21 App.jsx
    files. Defer unless the CSS load actually shows up on a
    Lighthouse run.

## Feature ideas: surfaces for uncovering market inefficiencies

These are practical additions consistent with the data Eric already
has under contract (Massive Options Starter and Indices Starter and
Stocks Starter), the existing model coverage on the site, and Eric's
documented preferences (no flow / tape-reading lab per memory
`feedback_no_flow_or_tape_lab`, no naive market-cap-weighted views per
memory `project_heatmap_design`). The ones that require a vendor tier
upgrade are flagged.

### Immediate (no new data needed)

- **Vol-surface arbitrage scanner.** A per-expiration panel that
  automatically flags butterfly violations (negative
  Breeden-Litzenberger density across consecutive strikes), calendar
  arbitrage (longer-expiry total variance below shorter-expiry at
  the same strike), and vertical-spread arbitrage (call or put
  spread cost exceeding the strike differential). The math is
  already in the smile and local pages; pulling out the explicit
  violations into a single screen turns existing diagnostics into a
  trade-prompting tool. Reader value: "here are strikes the market
  isn't pricing self-consistently right now."

- **0DTE vs front-week IV divergence history.** Daily plot of
  (0DTE ATM IV) minus (next-listed-expiration ATM IV) with a
  rolling percentile band. The 0DTE complex regularly gets unhinged
  from the rest of the front in both directions (squeeze mornings,
  pinned afternoons), and a historical chart of the spread surfaces
  those moments in advance of the day the dislocation matters.
  Already has the data: `daily_term_structure` carries both points.

- **Skew-percentile heatmap across single names.** Top 100 by
  options volume, current 25-delta put-call skew vs each name's own
  one-year history, sorted by extremity. The /heatmap page is
  same-day percent-change; this is the same equal-tile layout with
  IV-skew percentile coloring instead. Surfaces names whose skew is
  in the top or bottom 5% of their own distribution, which is where
  vol mispricings against component flows tend to live. Uses the
  options-volume roster the heatmap already has.

- **Vol-of-vol regime alert.** VVIX/VIX ratio crossing the existing
  yellow/orange/red thresholds (5/6/7) is already on /vix; what's
  missing is a forward-return histogram conditioned on the regime
  band. "When VVIX/VIX has been here historically, the next 5/10/20
  trading days returned what distribution?" Adds the conditional-
  expectations layer to the existing classifier.

- **Implied correlation premium dashboard.** SPX-implied vol decomposes
  into a sum of component vols weighted by index weights plus a
  correlation term. With the top-30 weights and their listed IVs
  (Stocks Starter covers all the heavyweights), you can extract the
  market-implied correlation each day and compare it to the realized
  correlation observed in the prior N days. The spread is the
  CIX-style implied correlation premium. Surfaces moments when the
  market is paying too much (or too little) for the diversification
  that index options imply. Pure derivative computation from data
  already in `daily_eod` + `daily_term_structure`.

- **Earnings-IV-crush calculator on the /earnings page.** /earnings
  already shows the implied move from the ATM straddle for each
  upcoming name. Adding a "historical move distribution" overlay
  for the same name over its past N earnings prints turns the
  implied move into a buy-or-sell-vol signal: names that
  consistently realize less than the implied move are systemically
  overpriced ahead of earnings. Needs ~8 quarters of historical
  daily aggregates per name (already covered by Stocks Starter for
  the heavyweights, more spotty for the long tail).

- **Cross-asset vol dislocation panel.** /vix has the cross-asset
  comparison framework. A dedicated panel that plots VIX,
  OVX (oil), MOVE (rates, if obtainable; otherwise GVZ for gold),
  and VVIX on a rolling z-score with the spread between them
  highlighted would surface moments when equity vol is leading or
  lagging the broader vol complex. The Nations skew/tail-cost
  cross-asset extension already in `vix_family_eod` covers the
  hardest part of the data acquisition.

- **GEX-based pinning probability for Friday opex days.** SPX dealer
  gamma concentration near spot near 16:00 ET on monthly opex
  Fridays correlates strongly with realized closing-print stickiness.
  The data is on the homepage already; what's missing is a
  conditional historical chart: "on Friday afternoons when GEX
  concentration within +/-0.5% of spot exceeded threshold X, the
  closing-print drift from the 14:00 ET print averaged Y." A
  Friday-only diagnostic surfaces a real trade.

### Requires data work but doable

- **Single-name dealer-gamma analogue for the top 30 by options
  volume.** The current SPX surfaces (gamma profile, vol flip,
  call/put walls) are exactly the kind of thing single-name traders
  ask for on NVDA, TSLA, MSFT, etc. Massive Stocks Starter doesn't
  include the options chain for those names, only daily aggregates;
  this would require an Options Starter expansion to include
  single-name chains at the per-strike level. The Options Starter
  tier already covers the OPRA feed for those names — verify before
  committing to the build.

- **Volatility carry tracker.** Daily plot of (long-dated minus
  short-dated SPX IV) divided by realized vol of that spread. The
  carry is the implicit funding cost of vol-long calendar trades.
  Useful for sizing the long-vol leg of any structural-hedge book.
  Plain derivation from `daily_term_structure`.

- **Stat-arb signal layer over `/rotations`.** The existing
  Mansfield-normalized rotation plane is a regime-detection surface.
  Adding a "mean-reversion candidate" filter — sectors whose rotation
  position diverges from their 60-day average by more than 2
  standard deviations — turns the page into a trade screen. Uses
  data already in `daily_eod`.

### Requires vendor tier upgrade (flag for later)

- **Microstructure-aware liquidity heatmap.** Bid-ask spreads on key
  strikes for the top-options-volume single names. The current
  Massive tier doesn't include synchronous quote-level data on
  individual options. Options Advanced ($199) does but is flagged
  in memory `reference_polygon_renamed_to_massive` as
  regulatorily-blocked for public sites.

- **MM positioning P&L estimator.** Combines GEX with intraday spot
  path to infer rough dealer P&L from delta-hedging. The reader-
  facing version is "when dealers are getting paid by gamma, the
  tape is quiet; when they're paying, the tape is choppy." Requires
  intraday quote-level data the current tier doesn't include.

## Areas where this audit said "the code is right, don't touch it"

For the record so a future audit doesn't relitigate:

- `usePlotly.js` is correctly self-hosting the cartesian Plotly bundle
  with preload + lazy `<script>` injection + module-singleton promise
  caching. The 1.3MB unminified file ships from `public/vendor/`
  exactly as documented in `index.html`.
- `data.mjs`, `vix-data.mjs`, `gex-history.mjs`, `vrp-history.mjs`
  all carry `Cache-Control` headers with appropriate s-maxage. The
  Netlify-functions agent flagged this as missing; it's not.
- `PageNarrator.jsx`'s polling useEffect depends on `fetchNarrative`
  which depends on `page`, so a page change correctly tears down the
  interval and starts a new one. The frontend agent flagged a stale-
  closure bug; there isn't one.
- `GammaInflectionChart.jsx`'s useEffect dependency array lists
  `hasProfile`, `split`, `dataMin`, `dataMax` alongside `profile`.
  All four are deterministically derived from `profile` so the list
  is verbose but correct, and `react-hooks/exhaustive-deps` would
  flag removal.
- Backfill scripts in `scripts/backfill/` are reference material
  Eric wants kept per memory
  `feedback_no_delete_backfill_without_confirm`. None of them are
  candidates for deletion.
- `AGENTS.md` and `CLAUDE.md` overlap intentionally — one is agent
  constraints, the other is codebase architecture. Both serve.
